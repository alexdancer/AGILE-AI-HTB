## Context

`docs/REACT_PORTAL_PARITY_PLAN.md` Phase 2. Phase 1
(`react-portal-default-rollback`) rolled `/`/login/logout back to Jinja because
the React shell is an incomplete surface. The React shell still exists at
`/app` but its layout is a centered topbar with flat nav links — a different
product from the Jinja Portal's full app frame in `base.html`.

Current React layout (`frontend/src/components/Shell.jsx`):

```
┌─────────────────────────────────────────────┐
│ brand          | Dashboard Projects Setup … │  topbar
├─────────────────────────────────────────────┤
│                                             │
│  centered .content (max-width 1200px)       │
│                                             │
└─────────────────────────────────────────────┘
```

Jinja layout (`src/agile_ai_htb/templates/base.html`):

```
┌─────────────────────────────────────────────┐
│ AGILE-AI-HTB · Portal            (topbar)  │
├──────────┬──────────────────────────────────┤
│ Projects │                                  │
│  · proj  │  .main (full width)              │
│  · + …   │                                  │
│ Setup    │                                  │
│ Govern.  │                                  │
│ Settings │                                  │
│ logout   │                                  │
├──────────┴──────────────────────────────────┤
│ AGILE-AI-HTB portal · … (footer)            │
└─────────────────────────────────────────────┘
```

The sidebar is generated by `portal_template_context` in
`template_context.py`: `{portal_auth_required, sidebar_projects:[{id,name,task_count}]}`.
The sidebar groups in `base.html` are static (Setup, Governance, Planning shown only when no projects, Settings) plus the dynamic project list.

This change makes the React shell render the same frame from the same data
source. No routing default change, no dashboard, no board workflow change.

## Goals / Non-Goals

**Goals:**

- The React shell presents the same app frame as Jinja: top brand bar, left
  sidebar, main full-width content, bottom footer.
- The sidebar lists connected projects (`Task board`/`No tasks`), shows the
  `└ Task board` sub-link when a project has tasks, and renders the Setup,
  Governance, Planning (no-projects-only), Settings groups + logout form,
  matching `base.html`.
- Active nav / active project is highlighted in the sidebar so the operator
  can tell where they are.
- Non-migrated Jinja pages are reachable from the React sidebar via ordinary
  full-page anchors; React-owned routes keep client-side navigation.
- The sidebar uses the same data source as Jinja via a new authenticated
  read-only JSON endpoint that reuses `portal_template_context` — no parallel
  counts, no new domain logic.

**Non-Goals:**

- No re-enable of React as the default landing (Phase 6 job).
- No dashboard, AGILE Board workflow, budget, review, or launch changes.
- No removal or rename of any Jinja route, declared React route, or JSON
  endpoint. The undeclared `/app/{path}` catch-all behavior is intentionally
  removed so unknown and extra-segment paths return 404.
- No new UI link to `/app` from the default Jinja Portal.
- No websockets, live project updates, or streaming Project list.
- No extraction of `tokens.css` and `base.html` into a single shared source in
  this change — `tokens.css` keeps mirroring `base.html` and the proposal notes
  this explicitly as a later cleanup.

## Decisions

### Decision 1: Reuse `portal_template_context`, do not duplicate the sidebar helper

Add one new GET endpoint `/api/portal/nav` in `react_shell.py` that returns
`{portal_auth_required, sidebar_projects}` by calling
`portal_template_context(request)` and trimming any keys the sidebar does not
need.

**Rationale:** `portal_template_context` is the existing single source for
sidebar projects (`id`, `name`, `task_count`) and the auth flag. Duplicating
the task-count loop in React-side code or a new helper would recreate a second
source of truth that can drift from Jinja. `base.html` already consumes this
context; React now consumes the same JSON.

**Alternatives considered:**
- *Reuse `/api/projects` instead.* Rejected: `/api/projects` returns the
  project-picker payload (`counts`, `total_tasks`, `capability`, `root_path`)
  shaped for the Home project cards, not the sidebar contract (`id`, `name`,
  `task_count`). Two shapes for the same concept would drift.
- *Inline the counts in the React component via fetch + deduping.* Rejected:
  forces the React component to know task-counting and auth-flag concerns the
  backend already owns.

### Decision 2: Sidebar link types — client-side vs full-page anchors

React-owned routes (`/app`, `/app/projects/:id`,
`/app/projects/:id/board`) use `AppLink` (client-side pushState via `NavContext`).
Non-migrated Jinja routes (`/dashboard`, `/projects`, `/setup`,
`/settings/control-plane`, `/settings/budget`, `/settings/project`,
`/settings/workers`, `/sessions`, `/alarms`, `/board`, `/logout`) render
ordinary `<a href>` tags so the browser does a full navigation to the Jinja
page — identical to today's topbar links.

**Rationale:** matches the existing convention in `Shell.jsx` and the plan's
implementation rule: "links to non-migrated Jinja surfaces should remain normal
full-page anchors until those surfaces are migrated." A missed full page load
to a non-migrated Jinja surface would leave the operator looking at a stale
React frame with no Jinja chrome behind it.

### Decision 3: Active-state derivation

Pass `activeView` and `activeProjectId` from `App.jsx` (where `parseRoute`
already runs) into `Shell.jsx`:
- `activeView: "home" | "workspace" | "board"` → highlight `Projects` group /
  `+ Open local repo` only when no project is active.
- `activeProjectId` → highlight the sidebar project item on workspace and board
  routes. Highlight its `└ Task board` sub-link only when `activeView` is
  `"board"`; the workspace keeps that subordinate action inactive.

Group items (`Dashboard`, `Sessions`, etc.) are never active in React because
those routes are non-migrated full-page Jinja pages — the React sidebar is
replaced on full load, so an "active" highlight would only flash. This matches
Jinja, where `active_page` is set by the Jinja route handler and the React
frame is gone the moment the browser lands on Jinja.

**Rationale:** keeping parity with Jinja's visual contract while not inventing
"active group" state React does not own.

### Decision 4: Mirror Jinja CSS values with an explicit React footer row

Add `.shell`, `.topbar`, `.sidebar`, `.group`, `.project-list`,
`.project-item`, `.project-name`, `.project-sub`, `.project-board`,
`.sidebar-action`, `.sidebar-empty`, `.logout`, `.main`, and `.shell-footer`
rules to `frontend/src/tokens.css`, preserving the visual values from
`base.html`. React keeps the footer inside `.shell`, so its grid intentionally
uses `auto 1fr auto` rows rather than Jinja's external-footer `auto 1fr` rows.
The existing shared `<style> :root` variables already match.

**Rationale:** `tokens.css` already mirrors `base.html`'s `:root`. Putting
layout here keeps "same visual product" declarative and reviewable in one
file. A later slice can extract a single shared source once both surfaces are
migrated; doing it now is horizontal cleanup outside this phase.

**Alternatives considered:**
- *CSS-in-JS.* Rejected: no styled-components/Emotion dependency in the frontend; plain CSS is the existing pattern.
- *Tailwind/config.* Rejected: introduces a new tool for a layout copy.

### Decision 5: Reject unknown `/app/*` routes

FastAPI serves the shell only for `/app`, `/app/projects/{id}`, and
`/app/projects/{id}/board`. Other paths under `/app` return 404, and the client
parser treats extra segments as `notFound` rather than silently rendering Home.

**Rationale:** a broad catch-all masks broken links, typos, and future
non-migrated routes. Explicit ownership keeps the React/Jinja boundary visible.

## Risks / Trade-offs

- **[Risk] `tokens.css` and `base.html` drift on future Jinja style edits.**
  → Mitigation: a frontend source-contract test asserts the sidebar group
  labels and `+ Open local repo` link render in the built shell. A separate
  color-token drift assertion already lives in the existing test suite
  through `tokens.css` comments; further syncing is a later cleanup slice.
- **[Risk] New endpoint leaks project names to unauthenticated callers.**
  → Mitigation: endpoint uses `Depends(require_portal_auth)` identical to
  `/api/projects`; a test asserts 401/403 when auth is required and no cookie is
  present.
- **[Risk] Sidebar project shape diverges from Jinja's contract.**
  → Mitigation: the endpoint derives from `portal_template_context`, so the
  data is literally the same dict; a shape test asserts `id`, `name`,
  `task_count` per project and the `portal_auth_required` boolean.
- **[Trade-off] No active highlight on Setup/Governance/Settings group items
  from React.** Accepted: those are non-migrated Jinja pages and the React
  frame is replaced on full navigation, matching Jinja's active-state lifetime.
- **[Trade-off] `tokens.css` and `base.html` keep duplicating layout rules.**
  Accepted: a shared CSS extraction is a later slice; doing it here would mix
  horizontal cleanup into a vertical parity slice.
